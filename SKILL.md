# Keti

以项目事实为依据维护科研课题，不把计划、证据和结论混在一起。先建立任务书，再执行；每个节点先获得用户确认，完成后汇报产物并等待用户验收；只有验收通过才解锁下游。

## 读取资源

- 初始化或修改课题时，读取 [references/workflow.md](references/workflow.md)。
- 创建或审查任务书时，读取 [references/task-contract.md](references/task-contract.md)。
- 登记图表、文献、实验指标或检查报告/PPT真实性时，读取 [references/evidence-contract.md](references/evidence-contract.md)。
- 需要调用其他 Skill 时，读取 [references/integrations.md](references/integrations.md)。
- 需要 PPT 模板、图片、字体或其他静态资源时，读取 [references/resources.md](references/resources.md)。
- 使用 `assets/` 中的模板作为起点；不要改写 Skill 自身模板。

## 核心工作流

1. 定位项目根目录。优先使用用户指定目录；否则使用当前工作目录。把所有运行产物写入课题项目，不写入本 Skill 目录。
2. 检查 `project.yaml`。若不存在，收集课题名称、方向、专业信息、导师信息与偏好、已有资料位置、截止时间及约束。列出并让用户勾定最终交付物，例如：项目/基金申请书、开题报告、中期报告、结题报告、论文、代码、实验数据与图表、答辩 PPT、逐页讲稿。**必须询问“本课题是否需要生成或修改代码？”**，除非用户已经明确说明。把答案记录为 `capabilities.code_generation: true|false`；不得根据专业方向自行推断。其他未知字段明确记为 `unknown`，仅追问会阻塞当前阶段的字段。
3. 按用户选择运行环境探针：
   `python <skill-dir>/scripts/probe_environment.py --project-root <project-root> --code-generation <yes|no>`。
   把缺失能力分为 `required`、`optional`、`unavailable`，给出降级路径；不得因可选依赖缺失阻塞规划。
4. 初始化新项目时运行：
   `python <skill-dir>/scripts/init_project.py --root <project-root> --title <title> --code-generation <yes|no>`。
   已有项目只补齐缺失结构，禁止覆盖用户文件；执行前用 `--dry-run` 预览。
5. 盘点本地资料、代码、数据、模板和云端链接。建立 `sources/source_registry.json`。云端资料必须先获取可访问内容；链接本身不是证据。
6. 制定检索计划。优先权威论文库、开放 API、出版社页面和专业组织；记录查询词、日期、来源及访问状态。遇到登录、验证码、付费墙或版权限制时停止绕过，改用摘要、开放版本或请用户提供合法副本。
7. 生成 `master_task.md` 和 `tasks/*.md`。任务粒度按实际课题与已选交付物裁剪，不强制生成无关模块。每一种用户要求的最终成果都必须有独立任务节点，不能用 PPT 大纲或汇报稿替代申请书、项目书、报告或论文。例如用户选择“项目申请 + PPT”，DAG 中必须同时存在“申请书撰写与审定”和“申请答辩 PPT”节点，且 PPT 依赖已审定的申请书。若 `code_generation` 为 `false`，不得创建代码架构、实现、构建、单元测试等任务，也不得为这些能力安装依赖；实验任务改为依赖用户已有数据、仪器记录或既有软件输出。每个任务必须声明依赖、输入、步骤、交付物、机器可验证验收规则、证据、状态、用户验收和变更历史。
8. **对每个节点设置强制执行确认门。**依赖满足后先向用户显示“节点执行卡”，包括：节点名称与目标、计划操作、将读取的输入、预计生成/修改的文件、是否调用外部 Skill、可能的额度/时间消耗、需要用户决定的问题。然后明确询问“是否执行当前节点？”。未收到针对当前节点的明确确认时，保持 `ready`，不得开始执行，也不得把对上一个节点的确认视为对后续节点的批量授权。一次确认默认只授权一个节点。
9. 对申请书/项目书、代码、实验三类高修改成本节点使用双重确认：第一次确认任务方案、章节/接口/实验设计与验收标准；第二次确认才执行正文写作、批量编码或实验运行。用户可要求先生成小样、单节、最小实现或试运行；不得默认连续生产全部内容。
10. 获得当前节点确认后把状态置为 `in_progress` 并执行。完成后运行：
   `python <skill-dir>/scripts/verify_deliverables.py --task <task-file> --project-root <project-root>`。
   自动验证通过后先置为 `awaiting_review`，不能直接置为 `completed`。验证失败时记录实际错误和下一步；任何会产生明显额度消耗的修复尝试都要再次征得用户确认，最多连续三次，仍失败则置为 `blocked` 并说明所需输入或权限。
11. **每个节点结束后必须发送节点完成报告。**报告至少包含：完成内容、验证结果、重要文件清单、证据边界、遗留问题和下一节点。重要文件必须逐项给出可点击的真实本地路径，并按类型归类：任务书/申请书/报告等 Markdown、代码、原始数据与指标、图表与 Archify 源文件、PPT/讲稿、日志。即使界面另有“已编辑文件”列表，也不能省略此报告。文件不存在时不得声称已生成。
12. 节点完成报告末尾必须询问用户选择：“验收通过并解锁下一节点 / 修改当前节点 / 暂停”。只有用户明确验收通过后，才把任务置为 `completed` 并展示下一节点执行卡；不得自动进入下一节点。
13. 需要代码时，代码与计算实验必须输出可复现命令、原始日志、结构化指标和图表生成脚本；不需要代码时，保留数据采集方法、原始记录、处理过程及人工操作说明。禁止把预期结果写成实测结果。报告和 PPT 只能使用清单中已登记且本地存在的真实资产。
14. 收到导师反馈或需求变更时，先更新直接受影响任务，再运行：
    `python <skill-dir>/scripts/invalidate_tasks.py --project-root <project-root> --changed <task-id> --apply`。
    先向用户汇报失效范围和预计成本，确认后仅重做传递下游；纯排版变更不得重跑实验，算法或实验设计变更必须使相关申请书、报告与 PPT 失效。每个重做节点仍须独立确认和验收。
15. 汇报阶段优先调用可用的 PPT Master Skill，并传入真实模板、已验证图表和逐页讲稿要求；无该 Skill 时生成逐页 Markdown 大纲与 speaker notes，不伪造 `.pptx`。PPT 节点不得替代其依赖的申请书、项目书或报告正文节点。
16. 最终汇报已完成、待审阅、阻塞、失效和待用户决策的任务，以及全部关键产物的真实路径。

## 资源发现与选择

按以下优先级解析 PPT 模板、图片、字体和其他静态资源：

1. 用户在当前提示词中明确指定的文件或目录。
2. 当前课题的 `templates/`、`assets/` 以及 `project.yaml.resources`。
3. 用户显式提供的额外资源根目录。
4. Keti 本地缓存 `.keti-resource-cache/packs/` 或用户级缓存目录。
5. `assets/resource-catalog.json` 中登记的可选远程资源包。

只扫描以上明确范围，不递归扫描整个主目录或磁盘。先运行 `scripts/discover_resources.py` 生成候选清单，并向用户展示资源名称、类型、来源、优先级、许可证状态和预览；选择后才挂载到当前任务。用户指定资源始终覆盖默认资源。

核心 Skill 仓库不得直接存放大型 `.pptx`、字体压缩包或成套图片。资源包必须版本化，包含 `pack.json`、许可证/归属信息和可选 SHA-256；远程资源必须先展示下载大小、来源与许可证，得到用户确认后才能下载到缓存。许可证为 `unknown` 的资源只能本地使用，不得自动发布、复制到交付物或上传仓库。

## 节点交互格式

执行前使用以下紧凑格式并等待回答：

```markdown
### 待确认节点：<ID> <名称>
- 目标：...
- 输入：`...`
- 本次操作：...
- 预计产物：`...md`、`...`
- 成本与风险：低/中/高；说明主要消耗与不确定性
- 待决定：...

是否执行当前节点？
```

执行后使用以下格式并等待验收：

```markdown
### 节点完成：<ID> <名称>
- 状态：`awaiting_review`
- 完成内容：...
- 验证：通过/未通过；说明证据
- 重要文件：
  - 申请书/报告：[文件名](/absolute/path/file.md)
  - 代码：[文件名](/absolute/path/file.py)
  - 数据与指标：[文件名](/absolute/path/metrics.json)
  - 图表：[文件名](/absolute/path/chart.svg)
- 遗留问题：...
- 下一节点：...

请选择：验收通过并解锁下一节点 / 修改当前节点 / 暂停。
```

## 强制规则

- 保留用户已有文件与未提交修改；所有会覆盖、下载、安装、联网写入或发布的动作均遵循当前环境授权。
- 不静默下载或安装第三方 Skill。先扫描当前已提供的 Skill；能力缺口再提出候选、来源、许可证、权限和风险，获得授权后使用受支持的安装机制。
- 不把搜索摘要当作已核验事实。每个外部事实保留可追溯引用；每个图表区分 `external_reference`、`generated_diagram` 与 `local_experiment`。
- 不以固定文件大小或“300 DPI”替代内容验证。图片验收同时检查存在性、可读性、来源、生成方式和任务相关性。
- 不在证据缺失时生成占位数据、虚构引用、虚构实验结果或假图。
- 不将“继续”“开始下一步”等模糊指令解释为多个节点的批量授权；展示当前节点执行卡后，只执行该节点。
- 不因已经生成 PPT 而判定课题成果完整；逐项核对 `project.yaml.deliverables` 和 `master_task.md`，缺少任何选定成果都必须保持项目未完成。

## 推荐任务 DAG

需要代码时可选择：`literature -> architecture -> implementation -> experiments -> application/report -> presentation`。不需要代码时使用裁剪链：`literature -> methodology/data collection -> analysis -> application/report -> presentation`。其中 `application/report` 必须展开为用户实际选择的申请书、项目书、开题/中期/结题报告或论文节点。允许文献与资料盘点并行；申请书和报告依赖已验证证据；演示依赖已验收的正文成果和已验证资产。所有箭头都表示“完成汇报并经用户验收后才可解锁”，不是自动连续执行。
